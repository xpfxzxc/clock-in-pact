# Readable User Guide

## 目标

将零散资料转成一份让普通用户愿意读、读得懂、用得上的使用指南。
始终优先帮助用户尽快完成第一次成功操作，再逐步建立整体理解。

## 执行流程

### 1) 建立事实基线

按以下顺序收集信息，并记录每条关键结论的来源文件：

1. 先读用户明确提供的材料
2. 再读 `README.md` 与相关文档（如 `docs/`）
3. 若仍不足，则扫描整个项目并补齐关键信息

信息不足时，明确标注“缺失信息”，不要臆测产品行为。

### 2) 定义读者与首个成功场景

先明确：

1. 指南给哪类“使用者”阅读
2. 使用者最想先完成的一个任务（First Win）

围绕这个任务组织开篇与前半部分内容，避免先讲概念再讲操作。

### 3) 输出文档主结构

保持连续叙事，避免模板化堆砌。
推荐结构（可按产品特性微调）：

1. 简短概要（说明这款应用解决什么问题，控制在短段落）
2. 立即上手（直接给最短路径，让用户几分钟内跑通）
3. 故事主线（以“一个用户从开始到稳定使用”的过程串联核心功能）
4. 关键补充（常见误区、边界条件、失败恢复、常见问题）
5. 总览收束（用文字 + 1~多张高信息密度配图快速复盘全局）

### 4) 写作约束

始终遵守以下规则：

1. 面向用户写作，不写开发者实现细节
2. 用日常语言，避免晦涩词和专业黑话
3. 每段尽量回答“用户下一步该做什么”
4. 优先给操作路径与判断标准，不只描述功能
5. 保持逻辑递进：先能用，再用好，再理解全貌

### 5) 图文与视频策略

仅在能提升理解时才加入配图或视频，不为“看起来丰富”而填充。

#### 配图策略

1. 优先选择高信息密度图：流程图、决策图、页面地图、状态转移图
2. 图必须服务于文字，补足文字难表达的结构关系
3. 若需要用户用 AI 生成配图，输出完整提示词

为“手绘信息图助手”生成提示词时：

1. 不写风格描述（风格由系统提示固定）
2. 写清主题、构图、信息层级、标签文案、重点标注关系
3. 仅给少量统一强调色建议
4. 使用 `references/prompts-and-templates.md` 的模板

#### 截图策略

需要截图时，明确给出：

1. 复现步骤（从哪个入口开始）
2. 截图时机（在哪一步、看到什么状态时截）
3. 截图目的（这张图帮助理解哪一段文字）

#### 视频策略

需要录屏时，明确给出：

1. 演示目标（用户看完能完成什么）
2. 演示步骤（按时间顺序）
3. 建议时长（尽量短，聚焦关键路径）
4. 每一步的讲解要点

### 6) 交付与占位

输出最终 Markdown 指南时：

1. 先给可直接阅读的正文
2. 对未提供的图片/视频，用清晰占位符标记插入点
3. 在文末集中列出“待补素材清单”

使用 `references/prompts-and-templates.md` 中的模板生成：

1. AI 配图提示词
2. 截图任务单
3. 视频录制任务单

## 质量检查

交付前逐项自检：

1. 开头是否简短后立刻进入上手操作
2. 是否有一条连续故事主线覆盖核心要点
3. 是否做到由浅入深、逻辑连贯
4. 是否明确区分“用户视角”与“开发者视角”
5. 所有结论是否能回溯到已读资料
6. 配图/视频是否高价值且不冗余
7. 是否给出完整素材请求（提示词、截图时机、录制步骤）

## 输出风格

保持具体、可执行、可复现。
避免空泛口号、功能清单式罗列、以及“看起来完整但不可操作”的说明。
